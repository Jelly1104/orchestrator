# 미래전략실(ATO) Playbook

> 이 문서는 AI 에이전트가 미래전략실의 철학과 정체성을 이해하기 위한 맥락 문서다.
> 실제 작업 규칙은 `AI_CONTEXT.md`를 따른다.

---

## 1. 미래전략실이란

미래전략실(AI Transformation Office)은 메디씨앤씨의 핵심 서비스인 의사 커뮤니티(메디게이트)와 경력관리 플랫폼(H-Link)을 AI 중심으로 재설계하는 전사 혁신 조직이다.

우리는 레거시 구조와 순차적 업무 방식을 AI 기반 병렬 프로세스로 전환한다.

---

## 2. 두 축의 역할

미래전략실은 두 가지 상호보완적 역할로 구성된다.

**AX (AI Experience)**는 "어떤 문제를 AI로 풀 것인가"를 결정한다. 사용자와 사업의 문제를 정의하고, AI 기술을 실제 경험으로 변환한다. 도메인 지식과 UX 관점이 핵심이다.

**AB (AI Builder)**는 "어떤 AI 기술을 어떻게 만들 것인가"를 결정한다. 모델 설계, 데이터 품질, 평가 기준, API 구조를 담당한다. 기술적 정확성과 효율성이 핵심이다.

두 역할은 분리되어 있지 않다. AX가 정의한 문제를 AB가 기술로 구현하고, 그 결과가 다시 AX의 경험 설계에 반영되는 순환 구조다.

---

## 3. 핵심 원칙

**병렬이 기본이다.** 기획-디자인-개발-QA는 순차가 아니라 동시에 진행된다. 모든 팀이 단일 문서, 단일 스펙 위에서 함께 움직인다.

**AI First.** 모든 산출물은 AI가 1차 생성하고 사람이 검토한다. 문서, 코드, 디자인 모두 마찬가지다.

**빠른 실험, 빠른 폐기.** 프로토타입은 24시간 내 생성하고 48시간 내 의사결정한다. 완벽함보다 속도를 우선한다.

**데이터 기반 판단.** 감이 아니라 사용자 행동과 로그를 기준으로 결정한다.

**재사용 우선.** 이미 만든 AI Layer(요약, 검색, 분류, 추천)는 전사가 공유한다. 중복 개발을 피한다.

**레거시보다 효율.** "원래 하던 방식"이 아니라 "더 빠르고 정확한 방식"을 선택한다.

**로컬 환경에서도 AX 관점의 사용 경험을 유지**하기 위해,각 기능은 독립된 URL/화면으로 접근 가능해야 하며,개발자는 공통 인덱스에서 기능별 데모를 손쉽게 탐색할 수 있어야 한다.

---

## 4. 3-Layer 구조

우리의 작업 흐름은 세 단계로 이루어진다.

**Experience 단계**에서 AX가 사용자 문제를 정의하고 UX 흐름과 기능 요구사항을 만든다.

**Builder 단계**에서 AB가 모델 범위를 정하고 데이터를 설계하며 품질 기준과 API 스펙을 확정한다.

**Development 단계**에서 FE/BE가 기능을 개발하고 통합하며 배포와 모니터링을 수행한다.

이 세 단계는 순차가 아니다. Experience가 문제를 정의하는 동안 Builder는 기술 검토를 시작하고, Development는 인프라를 준비한다.

---

## 5. 에이전트에게 기대하는 관점

당신은 이 팀의 동료다. 코드를 작성할 때 다음을 고민하라.

AX 관점에서, "이 코드가 최종 사용자에게 전달되는 경험을 저해하지 않는가?"

AB 관점에서, "이 로직이 데이터 파이프라인의 효율성을 해치지 않는가?"

병렬 프로세스 관점에서, "이 변경이 다른 팀의 작업과 충돌하지 않는가? 문서와 코드가 동기화되어 있는가?"

---

## 6. 이 문서와 AI_CONTEXT의 관계

**Playbook**은 "왜 우리가 이렇게 일하는가"를 설명한다. 팀의 철학, 원칙, 구조적 맥락을 담고 있다.

**AI_CONTEXT**는 "어떻게 일해야 하는가"를 정의한다. 구체적인 실행 규칙, 금지사항, 기술 스택을 담고 있다.

에이전트는 Playbook을 통해 맥락을 이해하고, AI_CONTEXT를 통해 행동한다. 두 문서가 충돌할 경우 AI_CONTEXT의 최우선 원칙을 따른다.
