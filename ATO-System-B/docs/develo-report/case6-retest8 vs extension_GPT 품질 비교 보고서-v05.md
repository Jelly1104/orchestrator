# case6-retest8 vs extension_GPT 품질 비교 보고서 v05

> 작성일: 2026-01-05  
> 작성자: ATO-System-B 개발팀  
> 목적: 동일 PRD(case6-retest8)로 생성한 GPT 기반 구현 결과와 기존 extension 모드 산출물을 비교하고, 보완 방향을 제시한다.

---

## 1. 핵심 요약

| 관점 | 우세 | 근거 |
|------|------|------|
| 보안/안전 | 🏆 **extension** | SELECT * 차단, SDD 사전 정의 쿼리, DOMAIN_SCHEMA/DB_ACCESS_POLICY 준수 |
| 구현 일관성 | 🏆 **extension** | PRD→IA→Wire→SDD 흐름 일관, API/쿼리 설계 포함 |
| 화면 충실도 | ⚠️ GPT | 구조는 재현했으나 스타일/데이터/인터랙션 미흡 |
| 개발 편의 | ⚠️ 비등 | GPT는 React 스캐폴딩 바로 사용, extension은 사전 정의 설계로 리스크 낮음 |

**요약**: GPT 결과는 레이아웃 뼈대는 빠르게 나왔지만, 실제 데이터/차트/인터랙션/보안 가이드가 비어 있음. 품질·안전·일관성 측면에서 extension 산출물이 여전히 우위.

---

## 2. 비교 상세

### 2.1 산출물 구조
- **GPT (이번 실험)**: `OrchestratorValidationDashboard.tsx` + 전용 CSS로 Wireframe 섹션을 React로 단일 페이지화. 정적 텍스트/바/배지 중심, 차트/데이터 연동 없음.
- **extension (기존)**: IA/Wire/SDD에 사전 정의 쿼리·API·검증 지표 포함. DB 대용량 주의, LIMIT/타임아웃 명시.

### 2.2 화면 품질/기능
| 항목 | GPT | extension |
|------|-----|-----------|
| 헤더/탭/섹션 | 구현 | 구현 |
| 메트릭/배지 | 정적 값만, 스타일 최소 | 정리된 컴포넌트/지표 정의 |
| 차트 | 플레이스홀더 텍스트 | 차트 요구사항 명시(라이브러리/데이터 스펙) |
| 인터랙션 | 버튼/필터 동작 없음 | HITL 체크포인트·승인 흐름 설명 |
| 반응형 | 기본 그리드만 | 화면별 동작 설명 |

### 2.3 데이터/보안
| 항목 | GPT | extension |
|------|-----|-----------|
| 쿼리/데이터 매핑 | 없음 (정적 더미) | SDD에 사전 정의 쿼리, 컬럼 제한, LIMIT/타임아웃 |
| SELECT * 방지 | 미정 | 준수 |
| 민감정보 필터 | 미정 | 준수 |
| 도메인 스키마 준수 | 미정 | 준수 |

### 2.4 구현 준비도
- **GPT**: 프런트 스캐폴딩만 제공 → 차트 라이브러리, API 연동, 상태관리, 테스트 모두 추가 필요.
- **extension**: API/쿼리 정의와 검증 규칙까지 있어 바로 개발 착수 가능, 리스크 낮음.

---

## 3. 결함 및 리스크 (GPT 결과 기준)
- 실제 데이터 연동 부재: KPI/메트릭/로그 모두 하드코딩, 실시간성 없음.
- 차트 부재: 플레이스홀더 텍스트만 존재, 그래프 라이브러리/데이터 스키마 없음.
- 인터랙션 미구현: 필터, 새로고침, 클릭/툴팁 등 wireframe 요구사항 반영 안 됨.
- 보안/품질 가드 미적용: SELECT * 방지, LIMIT, 타임아웃, 민감 컬럼 필터링 등 정책 미기재.
- 테스트/검증 없음: 단위·스냅샷·E2E 테스트 공백.

---

## 4. 개선 권고 (단계별)
1) **설계/보안 스펙 반영**: extension 산출물의 SDD 쿼리·API 정의, DB_ACCESS_POLICY, DOMAIN_SCHEMA를 프런트/백엔드 요구사항에 명시.
2) **데이터 연동**: KPI/메트릭/로그/Phase 진행률을 API로 연결, 활성 사용자 등 “?” 제거.
3) **차트 구현**: Recharts 등으로 A1~A3, 시스템 메트릭 차트 적용. 데이터 스키마를 SDD와 맞춤.
4) **인터랙션**: 필터(회원유형, 기간, LIMIT/타임아웃) 동작, 카드/행 클릭, 툴팁, 새로고침 구현.
5) **테스트/품질**: 컴포넌트 스냅샷, 차트 렌더 테스트, API 모킹, 접근성 체크.
6) **스타일/반응형 다듬기**: 토큰(색/간격/타이포) 정리, 모바일/태블릿 동작 점검.

---

## 5. 결론
- 동일 PRD 기준, **extension 모드가 보안·일관성·구현 준비도에서 우위**이며, GPT 결과는 “빠른 뼈대” 수준에 머무름.
- GPT 경로를 유지하더라도, 위 개선 권고(데이터·차트·보안·인터랙션·테스트)를 빠르게 채워 넣어야 실사용 가능 품질에 도달함. extension 산출물을 참조해 리스크를 줄이는 것을 권장.
