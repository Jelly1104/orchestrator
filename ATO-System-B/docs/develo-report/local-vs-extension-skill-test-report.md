## 2. SKILL.md 스타일 비교

### 2.1 체크리스트 방식 (테스트 A) ❌

```markdown
## 필수 항목
- [ ] 화면 3개 이상
- [ ] ASCII 레이아웃 포함
- [ ] 컴포넌트 정의됨

## Skill Report 출력
✅ Skill: doc-quality v1.0
✅ 품질 점수: 85/100
```

**특징**: "이것 해라, 저것 해라" 지시
**결과**: 최소 기준만 충족

### 2.2 ATO 스타일 (테스트 C, D) ✅

```markdown
## Identity
당신은 **Wireframe Designer Skill**입니다.
실제 개발자가 바로 구현할 수 있는 수준의 설계 문서를 만듭니다.

## Output Format (구체적 예시)
┌────────────────────────────────────────┐
│  [Logo] 시스템명     [알림🔔] [프로필👤] │
├────────────────────────────────────────┤
│  ┌─────────────┐ ┌─────────────┐       │
│  │  📊 KPI 1   │ │  📈 KPI 2   │       │
│  │   12,847    │ │    58,234   │       │
│  └─────────────┘ └─────────────┘       │
└────────────────────────────────────────┘

## 컴포넌트 정의 (예시)
| ID | 컴포넌트 | 타입 | Props |
|----|---------|------|-------|
| C1 | KPI 카드 | StatCard | value, trend |

## 좋은 예시 vs 나쁜 예시
✅ 좋은: 실제 데이터, 상세 레이아웃
❌ 나쁜: "차트 있음" 수준의 설명
```

**특징**: "이런 수준으로 만들어라" + 구체적 예시
**결과**: 예시 수준을 따라함

## 3. 테스트 D 핵심 발견: 명시적 Skill 참조 필요

### 3.1 문제점 1 (테스트 C): Report가 선택사항

```markdown
## Skill Report (선택)

작업 완료 시 아래 형식으로 요약 가능:
```

**결과**: Claude가 생략함 ❌

### 3.2 문제점 2 (테스트 D-1): 키워드만으로는 Skill 적용 안됨

**프롬프트**:
```
[Wireframe 설계] case6-retest11/PRD.md 기반으로 Wireframe을 만들어줘.
```

**결과**: Skill Report 없음 ❌ - **Skill 자체가 적용되지 않음**

### 3.3 해결책 (테스트 D-2): 명시적 SKILL.md 참조

**프롬프트**:
```
[Wireframe 설계] case6-retest11/PRD.md 기반으로 Wireframe을 만들어줘.

.claude/skills/doc-quality/SKILL.md를 참조해서 작성해줘.
```

**결과**: Skill Report 출력됨 ✅

### 3.4 Claude Code Skill 적용 조건

| 조건 | Skill 적용 | Report 출력 |
|------|-----------|-------------|
| 키워드만 (`[Wireframe 설계]`) | ❌ 보장 안됨 | ❌ |
| 명시적 참조 (`SKILL.md를 참조해서`) | ✅ 확실히 적용 | ✅ (필수 설정 시) |

**핵심**: Claude Code의 `.claude/skills/`는 **model-invoked** 방식이므로 명시적 요청이 필요

## 6. 로컬 vs Extension 방식 비교

### 6.1 방식별 특징

| 항목 | 로컬 (ATO-System-B) | Extension (Claude Code) |
|------|---------------------|-------------------------|
| Skill 위치 | `orchestrator/skills/` | `.claude/skills/` |
| 호출 방식 | 코드로 직접 호출 | Claude가 자동 선택 |
| 실행 보장 | ✅ 100% | ⚠️ 키워드 의존 |
| SKILL.md 효과 | 시스템 프롬프트로 주입 | Claude가 참조 |
| **권장 SKILL.md 스타일** | ATO 스타일 + 필수 Report | ATO 스타일 + 필수 Report |

### 6.2 결론

```
┌─────────────────────────────────────────────────────────────────────────────┐
│  SKILL.md 스타일이 방식보다 더 중요!                                         │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  로컬이든 Extension이든:                                                    │
│                                                                             │
│  ❌ 체크리스트 스타일 → 최소 기준만 맞춤                                     │
│  ⚠️ ATO 스타일 + Report(선택) → 풍부하지만 검증 불가                        │
│  ✅ ATO 스타일 + Report(필수) → 풍부함 + 검증 가능                          │
│                                                                             │
│  핵심: "무엇을 해라"가 아니라 "이렇게 해라"를 보여주고                       │
│        "반드시 Report를 출력해라"로 검증 가능하게                            │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

## 7. 권장사항

### 7.1 SKILL.md 작성 시

| 우선순위 | 권장 사항 | 이유 |
|---------|----------|------|
| **P0** | Identity 섹션 필수 | 역할 명확화 |
| **P0** | Output Format에 구체적 예시 | Claude가 수준 파악 |
| **P0** | 좋은/나쁜 예시 포함 | 품질 기준 명확화 |
| **P0** | Skill Report `(필수)` + "반드시" | 적용 여부 검증 ⚠️ NEW |
| P1 | Workflow 섹션 | 작업 순서 가이드 |

### 7.2 Skill Report 필수화 문구

```markdown
## Skill Report (필수)

⚠️ **반드시 출력 마지막에 아래 형식으로 Skill Report를 포함해야 합니다.**
이 리포트가 없으면 Skill이 적용되지 않은 것으로 간주됩니다.
```

### 7.3 프롬프트 작성 시

| 우선순위 | 권장 사항 | 예시 |
|---------|----------|------|
| **P0** | **명시적 SKILL.md 참조** ⚠️ | `.claude/skills/xxx/SKILL.md를 참조해서` |
| P1 | 트리거 키워드 포함 | `[Wireframe 설계]` |
| P2 | 구체적 요구사항 명시 | "화면 2개", "모바일 포함" |

### 7.4 권장 프롬프트 형식

```
[작업 유형] {요구사항}

.claude/skills/{skill-name}/SKILL.md를 참조해서 작성해줘.
```
